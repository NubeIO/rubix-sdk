# Manufacturing Run Implementation - Scope Document

## Overview

Manufacturing Runs track production batches of hardware products. A production run represents a batch order (e.g., "make 100 units of rev-c hardware") and contains individual produced units as children.

**Key Features:**
- ✅ Auto-generated run numbers with timestamps (e.g., `MR-2026:12:22-14:30`)
- ✅ Track target quantity vs. produced count
- ✅ Factory workflow for entering individual units
- ✅ Hardware version tracking per run
- ✅ Serial number assignment per unit
- ✅ RMA support for failed units
- ✅ Reuses `core.asset` for units (no custom type needed)

## Architecture

### Node Hierarchy

```
core.product (e.g., "Edge Controller Pro")
│
├─ plm.manufacturing-run (MR-2026:12:22-14:30)
│  │  runNumber: "MR-2026:12:22-14:30" (auto-generated)
│  │  hardwareVersion: "rev-c" (user enters)
│  │  targetQuantity: 100 (user enters)
│  │  producedCount: 47 (calculated from children)
│  │  status: "in-progress"
│  │
│  ├─ core.asset (Unit EC-C-1401)
│  │  │  serialNumber: "EC-C-1401"
│  │  │  hardwareRevision: "rev-c"
│  │  │  status: "produced"
│  │  │
│  │  └─ core.ticket (RMA-001)
│  │     ticketType: "rma"
│  │     issue: "Power failure"
│  │
│  ├─ core.asset (Unit EC-C-1402)
│  │  serialNumber: "EC-C-1402"
│  │  status: "shipped"
│  │
│  └─ core.asset (Unit EC-C-1403)
│     serialNumber: "EC-C-1403"
│     status: "qa-failed"
│
└─ plm.manufacturing-run (MR-2026:11:15-09:00)
   (previous run)
```

### Node Types Used

| Node Type | Purpose | Why This Type? |
|-----------|---------|----------------|
| `plm.manufacturing-run` | Production batch/order | Custom type - no core equivalent for manufacturing |
| `core.asset` | Individual produced units | Perfect fit - serialNumber, hardwareRevision, warranty |
| `core.ticket` | RMAs for failed units | Service tickets with issue tracking |

**Only 1 custom type needed:** `plm.manufacturing-run`

## Data Model

### plm.manufacturing-run Settings

```typescript
interface ManufacturingRunSettings {
  // Auto-generated by backend
  runNumber: string;              // "MR-2026:12:22-14:30"

  // User enters when creating run
  hardwareVersion: string;        // "rev-c", "v2.1", etc.
  targetQuantity: number;         // How many to produce (e.g., 100)
  facilityLocation?: string;      // "Factory A", "Shanghai Plant"

  // Tracking fields
  producedCount: number;          // Calculated from child assets
  status: 'planned' | 'in-progress' | 'qa' | 'completed' | 'cancelled';

  // Optional metadata
  serialRangeStart?: string;      // "EC-C-1401"
  serialRangeEnd?: string;        // "EC-C-1500"
  productionDate?: string;        // ISO date
  qaFailures?: number;            // Count of failed units
  batchNotes?: string;            // Free text notes
}
```

### core.asset Settings (Units)

```typescript
interface UnitSettings {
  // Core asset fields
  serialNumber: string;           // "EC-C-1402" (user enters)
  assetType: string;              // "edge-controller"
  hardwareRevision: string;       // "rev-c" (copied from run)
  status: 'produced' | 'qa-pass' | 'qa-fail' | 'shipped' | 'installed' | 'rma';

  // Optional manufacturing metadata
  productionRunNumber?: string;   // "MR-2026:12:22-14:30" (denormalized for display)
  manufactureDate?: string;       // ISO date
  qaDate?: string;                // ISO date
  qaStatus?: string;              // "pass", "fail", "pending"

  // Installation/warranty (filled later)
  firmwareVersion?: string;       // Installed firmware
  installDate?: string;           // When deployed
  warrantyUntil?: string;         // Warranty expiration
  location?: string;              // Where installed
}
```

## Factory Workflow

### 1. Manager Creates Production Run

**User Action:**
- Navigate to product detail page
- Click "New Production Run"
- Fill form:
  - Hardware Version: `rev-c`
  - Target Quantity: `100`
  - Facility: `Factory A`
- Click Save

**Backend:**
- Auto-generates `runNumber`: `MR-2026:12:22-14:30` (timestamp-based)
- Creates `plm.manufacturing-run` node under product
- Sets `producedCount: 0`, `status: 'planned'`

**Result:**
```
Run: MR-2026:12:22-14:30
Hardware: rev-c
Target: 100 units
Produced: 0/100 (0%)
Status: Planned
```

### 2. Factory Produces Units

#### Option A: Single Unit Entry

**User Action:**
- Open production run
- Click "+ Add Unit"
- Enter serial number: `EC-C-1402`
- (Optional) QA status, notes
- Click Save

**Backend:**
- Validates serial number is unique globally
- Creates `core.asset` node under run
- Copies `hardwareRevision` from run
- (Optional) Copies `runNumber` to settings for display
- Auto-increments run's `producedCount` (via hook)

#### Option B: Bulk CSV Import ⭐ (Phase 1 - Critical)

**User Action:**
- Open production run
- Click "Bulk Import Units"
- Upload CSV file:
  ```csv
  serialNumber,qaStatus,notes
  EC-C-1401,pass,
  EC-C-1402,pass,
  EC-C-1403,fail,Power supply issue
  ```
- Review preview (shows duplicates, errors)
- Click "Import"

**Backend:**
- Validates all serial numbers (uniqueness, format)
- Creates assets in batch
- Auto-updates `producedCount` and `qaFailures` via hooks

**Why critical:** Adding 100+ units one-by-one is unusable for real factory workflows.

**Result:**
```
Run: MR-2026:12:22-14:30
Produced: 47/100 (47%)
Status: In Progress

Units:
- EC-C-1401 (produced, QA pass)
- EC-C-1402 (produced, QA pending)
- ...
```

### 3. QA and Completion

**QA Process:**
- Factory marks each unit status: `qa-pass` or `qa-fail`
- Failed units tracked in run's `qaFailures` count

**Completion:**
- When `producedCount >= targetQuantity`
- Manager marks run status: `completed`

### 4. RMA Handling (Later)

**When unit fails in field:**
- Navigate to unit detail page
- Create RMA ticket (child of unit)
- Type: `core.ticket` with `ticketType: 'rma'`
- Track issue, resolution, replacement

### 3. Backend Hook (Auto-generate Run Number)

**File:** `nube.plm/backend/hooks/manufacturing_run.go`

```go
package hooks

import (
    "fmt"
    "time"
    "github.com/NubeIO/rubix/internal/services/hooks/hookctx"
)

type ManufacturingRunHook struct{}

func (h *ManufacturingRunHook) OnCreate(ctx *hookctx.Context) error {
    // Auto-generate runNumber if not provided
    if runNumber, ok := ctx.Node.Settings["runNumber"].(string); !ok || runNumber == "" {
        now := time.Now()
        runNumber := fmt.Sprintf("MR-%d:%02d:%02d-%02d:%02d",
            now.Year(),
            now.Month(),
            now.Day(),
            now.Hour(),
            now.Minute(),
        )
        ctx.Node.Settings["runNumber"] = runNumber
    }

    // Initialize counters
    if _, ok := ctx.Node.Settings["producedCount"]; !ok {
        ctx.Node.Settings["producedCount"] = 0
    }
    if _, ok := ctx.Node.Settings["qaFailures"]; !ok {
        ctx.Node.Settings["qaFailures"] = 0
    }

    return nil
}
```

**Register hook in plugin initialization.**

### 4. Auto-Update Mechanisms

**Problem:** `producedCount` and `qaFailures` must stay in sync with actual units.

**Solution:** Backend hooks that trigger when units are created/updated.

**File:** `nube.plm/backend/hooks/asset_tracking.go`

```go
package hooks

import "github.com/NubeIO/rubix/internal/services/hooks/hookctx"

type AssetTrackingHook struct{}

// OnCreate - When asset is added under a manufacturing run
func (h *AssetTrackingHook) OnCreate(ctx *hookctx.Context) error {
    if ctx.Node.Type != "core.asset" {
        return nil
    }

    // Check if parent is manufacturing run
    if ctx.Parent != nil && ctx.Parent.Type == "plm.manufacturing-run" {
        // Increment producedCount
        count := ctx.Parent.Settings["producedCount"].(float64)
        ctx.Parent.Settings["producedCount"] = count + 1

        // Update qaFailures if applicable
        if status, ok := ctx.Node.Settings["qaStatus"].(string); ok && status == "fail" {
            failures := ctx.Parent.Settings["qaFailures"].(float64)
            ctx.Parent.Settings["qaFailures"] = failures + 1
        }

        // Save parent
        return ctx.UpdateNode(ctx.Parent)
    }

    return nil
}

// OnUpdate - When asset QA status changes
func (h *AssetTrackingHook) OnUpdate(ctx *hookctx.Context) error {
    if ctx.Node.Type != "core.asset" || ctx.Parent == nil {
        return nil
    }

    if ctx.Parent.Type == "plm.manufacturing-run" {
        // Check if qaStatus changed
        oldStatus := ctx.OldNode.Settings["qaStatus"]
        newStatus := ctx.Node.Settings["qaStatus"]

        if oldStatus != newStatus {
            failures := ctx.Parent.Settings["qaFailures"].(float64)

            if newStatus == "fail" {
                failures++
            } else if oldStatus == "fail" {
                failures--
            }

            ctx.Parent.Settings["qaFailures"] = failures
            return ctx.UpdateNode(ctx.Parent)
        }
    }

    return nil
}

// OnDelete - When asset is removed
func (h *AssetTrackingHook) OnDelete(ctx *hookctx.Context) error {
    if ctx.Node.Type != "core.asset" || ctx.Parent == nil {
        return nil
    }

    if ctx.Parent.Type == "plm.manufacturing-run" {
        // Decrement producedCount
        count := ctx.Parent.Settings["producedCount"].(float64)
        ctx.Parent.Settings["producedCount"] = count - 1

        // Update qaFailures if needed
        if status, ok := ctx.Node.Settings["qaStatus"].(string); ok && status == "fail" {
            failures := ctx.Parent.Settings["qaFailures"].(float64)
            ctx.Parent.Settings["qaFailures"] = failures - 1
        }

        return ctx.UpdateNode(ctx.Parent)
    }

    return nil
}
```

```typescript
export function useProductionRun(runId: string) {
  const client = usePluginClient();

  // Get run details
  const runQuery = useQuery({
    queryKey: ['production-run', runId],
    queryFn: () => client.getNode(runId)
  });

  // Get units (children)
  const unitsQuery = useQuery({
    queryKey: ['production-run-units', runId],
    queryFn: async () => {
      const result = await client.queryNodes({
        filter: `type is "core.asset" and parent.id is "${runId}"`
      });
      return result.nodes || [];
    }
  });

  // Calculate QA metrics from units
  const qaMetrics = React.useMemo(() => {
    const units = unitsQuery.data || [];
    return {
      passed: units.filter(u => u.settings?.qaStatus === 'pass').length,
      failed: units.filter(u => u.settings?.qaStatus === 'fail').length,
      pending: units.filter(u => !u.settings?.qaStatus || u.settings.qaStatus === 'pending').length
    };
  }, [unitsQuery.data]);

  return {
    run: runQuery.data,
    units: unitsQuery.data || [],
    qaMetrics,
    isLoading: runQuery.isLoading || unitsQuery.isLoading
  };
}
```

## Validation & Constraints

### Serial Number Validation

**Uniqueness:** Serial numbers must be globally unique across all assets in the system.

**Backend validation:**
```go
func (h *AssetHook) OnCreate(ctx *hookctx.Context) error {
    if serialNumber, ok := ctx.Node.Settings["serialNumber"].(string); ok && serialNumber != "" {
        // Check for duplicates
        existing := ctx.QueryNodes(Filter{
            Type: "core.asset",
            Settings: map[string]interface{}{
                "serialNumber": serialNumber
            }
        })

        if len(existing) > 0 {
            return fmt.Errorf("serial number '%s' already exists", serialNumber)
        }

        // Optional: Validate format (e.g., "EC-C-####")
        if !regexp.MustCompile(`^[A-Z]{2}-[A-Z]-\d{4}$`).MatchString(serialNumber) {
            return fmt.Errorf("invalid serial format: must be XX-X-#### (e.g., EC-C-1401)")
        }
    }

    return nil
}
```

### Status Transition Rules

**Manufacturing Run:**
- `planned` → `in-progress`, `cancelled`
- `in-progress` → `qa`, `cancelled`
- `qa` → `completed`, `in-progress` (if re-testing)
- `completed` → (terminal, no changes)
- `cancelled` → (terminal, no changes)

**Units (core.asset):**
- `produced` → `qa-pass`, `qa-fail`
- `qa-pass` → `shipped`
- `qa-fail` → `rma` (with RMA ticket created)
- `shipped` → `installed`
- `installed` → `rma` (field failure)

### Delete Constraints

**Manufacturing Run:**
- ❌ Cannot delete if units exist (child assets)
- ✅ Can archive (set status to `cancelled`)
- ✅ Can delete if no children

**Units:**
- ✅ Can delete (decrements run's `producedCount` via hook)
- ⚠️ Warning if unit has RMA tickets

## Edge Cases & Error Handling

### Concurrent Updates
**Scenario:** Two workers add same serial number simultaneously.

**Solution:** Backend atomic check-and-create with unique constraint on `serialNumber` field.

### Bulk Import Errors
**Scenario:** CSV contains duplicates or invalid formats.

**Solution:**
- Pre-validate entire batch before import
- Show error report with line numbers
- Allow partial import (skip invalid rows) or reject all

### Run Deletion with Units
**Scenario:** Manager tries to delete run with 47 produced units.

**Solution:**
- Block deletion with error: "Cannot delete run with units. Archive or remove units first."
- Offer "Archive" button (sets `status: 'cancelled'`)

### QA Status Changes
**Scenario:** Unit marked as `qa-fail`, then re-tested and changed to `qa-pass`.

**Solution:**
- Backend hook detects status change
- Decrements `qaFailures` on run
- Audit log records who changed status and when

### Serial Range Conflicts
**Scenario:** `serialRangeEnd` is lower than `serialRangeStart`.

**Solution:**
- Frontend validation before submitting
- Backend validation: `rangeEnd >= rangeStart`

### producedCount Drift
**Scenario:** Manual DB edits or failed transactions cause count mismatch.

**Solution:**
- Add admin tool: "Recalculate Counts" (queries children and updates parent)
- Nightly job to verify counts match actual children

## Key Design Decisions

### ✅ Auto-generate Run Number
**Format:** `MR-YYYY:MM:DD-HH:MM`

**Why:**
- Chronological sorting
- Human-readable
- No user input needed
- Unique (timestamp-based)

**Alternative considered:** UUID-based (MR-U6JH567-26:03) - rejected as less readable

### ✅ Use core.asset for Units
**Why:**
- Already has `serialNumber`, `hardwareRevision`, `warranty` fields
- No custom type needed
- Reuses existing core functionality

**Alternative considered:** Custom `plm.unit` type - rejected as unnecessary

### ✅ Hierarchy vs. Refs
**Decision:** Use `parentRef` (hierarchy)

**Why:**
- Natural parent/child relationship (run contains units)
- Simple queries: `parent.id is "{runId}"`
- Units belong to exactly one run

**Optional enhancement:** Denormalize `runNumber` into unit settings for display performance

### ✅ Status Tracking
**Run statuses:**
- `planned` - Created but not started
- `in-progress` - Factory producing units
- `qa` - QA testing phase
- `completed` - All units produced and shipped
- `cancelled` - Run aborted

**Unit statuses:**
- `produced` - Manufactured
- `qa-pass` / `qa-fail` - QA results
- `shipped` - Sent to customer
- `installed` - Deployed in field
- `rma` - Returned for repair

### ✅ RMA as Children of Units
**Hierarchy:**
```
core.asset (Unit EC-C-1402)
└─ core.ticket (RMA for power failure)
```

**Why:**
- RMA is related to a specific unit
- Can track multiple RMAs per unit over lifetime
- Uses existing `core.ticket` type

### ✅ producedCount Calculation
**Decision:** Auto-updated by backend hooks, not manually by frontend.

**Why:**
- Single source of truth (backend)
- No risk of frontend forgetting to update
- Handles deletions/updates correctly
- Counts include all units regardless of QA status

**What it includes:**
- All child assets (produced, qa-pass, qa-fail, shipped, etc.)
- Does NOT exclude failed units (they were still produced)
- Separate `qaFailures` field tracks failures

### ✅ Bulk CSV Import Required for MVP
**Decision:** Include bulk CSV import in Phase 1, not Phase 2.

**Why:**
- Adding 100+ units one-by-one is unusable in real factory
- Will block testing with real production runs
- CSV is familiar format for factory workers
- Simple to implement (validate → batch create)

**Not in MVP:**

## Summary

Manufacturing Runs provide a complete production tracking solution:

- **Simple data model**: 1 custom type (`plm.manufacturing-run`) + reuse `core.asset`
- **Auto-generated IDs**: Timestamp-based run numbers (MR-2026:12:22-14:30)
- **Auto-updated counts**: Backend hooks maintain `producedCount` and `qaFailures`
- **Robust validation**: Serial uniqueness, status transitions, delete constraints
- **Factory workflow**: Manager creates run → factory adds units (single/CSV bulk) → track progress
- **Bulk CSV import**: Critical for usability with 100+ unit batches
- **Scalable**: Supports hundreds of runs with thousands of units
- **Extensible**: Ready for RMA tracking via child tickets

**Critical for Phase 1:**
- Backend auto-update hooks (producedCount sync)
- Serial number validation (uniqueness + format)
- Bulk CSV import (usability requirement)
